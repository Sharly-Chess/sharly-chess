# Developer Guide — Linux Flatpak CI/CD

This document is for **developers** contributing to the project. It explains the CI/CD architecture, automated workflows, and procedures to follow. **No local build is required** — everything runs through GitHub Actions.

---

## Overview

The `linux-flatpak` branch provides the Linux Flatpak distribution of Sharly Chess. It is integrated into the main `export.yml` workflow alongside the Windows and macOS builds.

### Branches

| Branch | Role |
|--------|------|
| `linux-flatpak` | Linux Flatpak branch. Contains Flatpak configuration and scripts |
| `dev` | Main development branch |
| `gh-pages` | Hosts the OSTree repository (`repo/`) via GitHub Pages |

### OSTree repository (GitHub Pages)

The OSTree repository is published to `repo/` on GitHub Pages at `https://flatpak.sharly-chess.com/repo/` on every tagged release.

## GitHub Actions workflow

### `export.yml` — Build & Publish (Linux job)

The Linux job in `export.yml` builds the Flatpak for both architectures and publishes to GitHub Pages.

```
                    ┌─────────────────┐
                    │    Trigger      │
                    │                 │
                    │ • tag push →    │
                    │   production    │
                    │ • manual        │
                    │   dispatch      │
                    └────────┬────────┘
                             │
                ┌────────────┴────────────┐
                ▼                         ▼
        ┌───────────────┐         ┌───────────────┐
        │  build-x86_64 │         │  build-arm64  │
        │  (ubuntu-24)  │         │  (ubuntu-24   │
        │               │         │   ARM runner) │
        │ 1. Checkout   │         │               │
        │ 2. Runtimes   │         │  (same steps) │
        │ 3. Secrets    │         │               │
        │ 4. Build      │         │               │
        │ 5. Bundle     │         │               │
        │ 6. Archive    │         │               │
        │    OSTree     │         │               │
        └───────┬───────┘         └───────┬───────┘
                │                         │
                └────────────┬────────────┘
                             │ artifacts
                             ▼
                ┌─────────────────────────┐
                │        publish          │
                │                         │
                │ 1. Checkout gh-pages    │
                │ 2. Download artifacts   │
                │ 3. Target: repo/        │
                │ 4. Import GPG           │
                │ 5. Merge OSTree repos   │
                │ 6. Sign commits         │
                │ 7. Static deltas        │
                │    (+ from-empty)       │
                │ 8. Generate .flatpakrepo│
                │ 9. Push gh-pages        │
                └─────────────────────────┘
```

#### Triggers

| Event | Example |
|-------|---------|
| Push tag `*` | Tag `3.5.2` → build + publish to `repo/` |
| Manual dispatch with `version` | Rebuild an existing tag |

> **Note:** Changes to `*.md` and `docs/**` files do not trigger a build.

#### Required GitHub secrets

| Secret                              | Description                                                        |
|-------------------------------------|--------------------------------------------------------------------|
| `GPG_PRIVATE_KEY`                   | Private GPG key for signing the OSTree repository                  |
| `FIDE_DATABASE_ENC_PASSWORD`        | FIDE database encryption password                                  |
| `FFE_SQL_SERVER_HOST`               | FFE SQL server host                                                |
| `FFE_SQL_SERVER_USER`               | FFE SQL server user                                                |
| `FFE_SQL_SERVER_PASSWORD`           | FFE SQL server password                                            |
| `FFE_SQL_SERVER_DATABASE`           | FFE SQL server database name                                       |
| `FFE_DATABASE_ENC_PASSWORD`         | FFE local database encryption password                             |
| `FFE_DATABASE_ZIP_PASSWORD`         | FFE local database zip password (to be removed in future versions) |
| `FRA_SCHOOLS_DATABASE_ENC_PASSWORD` | FRA Schools local database encryption password                     |
| `CHESS_RESULTS_AES_KEY`             | Chess-Results AES key                                              |
| `CHESS_RESULTS_AES_IV`              | Chess-Results AES IV                                               |

#### Optimisations

- **Static deltas**: Generated automatically by `flatpak build-update-repo --generate-static-deltas` for incremental updates.
- **Static deltas from-empty**: Generated by `ostree static-delta generate --empty` to speed up first installation (avoids thousands of small HTTP requests).
- **Prune**: `--prune` removes obsolete OSTree objects to limit repository size.
- **Commit history**: `flatpak build-commit-from` links each new commit to the previous one, enabling rollback and delta updates.

## Procedures

### Publish a new stable version

When a new release tag is pushed, the `export.yml` workflow runs automatically. The Linux job:

1. Builds the Flatpak for x86_64 and arm64
2. Publishes to `repo/` (production OSTree)
3. Uploads `.flatpak` bundles to the GitHub Release

**Manual rebuild** (if a build failed or a secret changed):

```
GitHub Actions → Sharly-Chess export → Run workflow
  version_tag: 3.5.2
```

### Test a pre-release

1. Push a pre-release tag (e.g. `3.6.0b1`)
2. The workflow builds and publishes to `repo/`
3. Test on a Linux machine:
   ```bash
   flatpak update --user   # if dev channel is installed
   ```

> **Note:** Data is not at risk: each version creates its own data folder (e.g. `sharly-chess-3.6.0b1/`).

### Modify the Flatpak configuration

1. Edit the files in `scripts/export/linux/flatpak/configuration/` on `linux-flatpak`
2. Push → the dev build triggers automatically
3. Check in GitHub Actions
4. If OK and intended for production → run a manual rebuild:
   ```
   GitHub Actions → Sharly-Chess export → Run workflow
     version_tag: <current_version>
   ```

## Flatpak configuration files

| File | Role |
|------|------|
| `scripts/export/linux/flatpak/configuration/com.sharlychess.SharlyChess.json` | Manifest (app-id, runtime, modules, build-commands) |
| `scripts/export/linux/flatpak/configuration/com.sharlychess.SharlyChess.desktop` | Desktop entry (name, icon, categories) |
| `scripts/export/linux/flatpak/configuration/com.sharlychess.SharlyChess.appdata.xml` | AppStream metadata (description, releases, screenshots) |
| `scripts/export/linux/flatpak/scripts/launcher.py` | Launch within the Flatpak sandbox |
| `scripts/export/linux/flatpak/scripts/generate_notice.py` | Third-party licence NOTICE generation |
| `scripts/export/linux/flatpak/scripts/sync_desktop_version.py` | Sync version in .desktop file |

### What to update and when

| Event | File(s) | Automated? |
|-------|---------|------------|
| New upstream version | `appdata.xml` (release notes) | Manual — update before tagging |
| New upstream version | `.desktop` (version) | ✅ Yes (`sync_desktop_version.py` in build) |
| GNOME runtime version change | Manifest JSON (`runtime-version`) | Manual |
| New Python dependency | `pyproject.toml` | Manual |
| Permission change | Manifest JSON (`finish-args`) | Manual |
| Python version change | Manifest JSON + `PYTHONPATH` | Manual |

## Flatpak permissions

Configured in the manifest (`finish-args`):

| Permission | Reason |
|-----------|--------|
| `--share=network` | Web service (Litestar/Uvicorn on port 8080) + FFE, FIDE, Chess-Results access |
| `--filesystem=home:rw` | User file import/export (Documents, Downloads) |
| `--socket=wayland` | GUI display (modern protocol) |
| `--socket=x11` | GUI display (compatibility) |
| `--socket=pulseaudio` | Sound notifications |
| `--device=dri` | GPU acceleration |
| `--env=PYTHONUNBUFFERED=1` | Real-time Python logs |
| `--env=PYTHONPATH=...` | Python module path |

## Data storage

Data is isolated per version within the Flatpak sandbox:

```
~/.var/app/com.sharlychess.SharlyChess/data/
└── sharly-chess-X.Y.Z/
    ├── events/          # Tournaments (.sce) and configuration (.scc)
    │   └── archives/    # Archived tournaments (.sca)
    ├── logs/            # Activity log
    ├── tmp/             # Temporary databases (FIDE, FFE, sessions)
    └── custom/          # Custom files
```

The `launcher.py` script configures `XDG_DATA_HOME`, creates the sub-folders, and sets the `CWD` to the version folder. The application then uses relative paths.

## Flathub (future)

Publishing to Flathub will require adaptations:

1. **No network access at build time**: Use `flatpak-pip-generator` to pre-declare all Python dependencies as offline sources
2. **Submission**: Fork `flathub/flathub`, branch `new-pr`, submit the adapted manifest
3. **Updates**: Via PR on the Flathub repository

For now, distribution is exclusively via the self-hosted repository on GitHub Pages.
